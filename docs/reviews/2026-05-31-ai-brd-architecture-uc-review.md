## Findings

### [P1] UC-06 requires BRD sections that the Phase 1 template does not define

- Evidence:
  - the canonical spec includes `loops[]` and `annotations[]` in [ai-brd-description-feature.md](../product/ai-brd-description-feature.md:369) and [ai-brd-description-feature.md](../product/ai-brd-description-feature.md:373).
  - the Phase 1 default BRD template only defines 10 sections and does not include either `Loops` or `Annotations` in [ai-brd-description-feature.md](../product/ai-brd-description-feature.md:387).
  - UC-06a expects “BRD output có section `Loops`” in [UC-06-sinh-brd-tu-diagram.md](../use-cases/UC-06-sinh-brd-tu-diagram.md:58).
  - UC-06c expects notes to render into an `Annotations` section in [UC-06-sinh-brd-tu-diagram.md](../use-cases/UC-06-sinh-brd-tu-diagram.md:71).
- Impact:
  - implementers do not know whether Phase 1 should add new BRD sections, fold loops/annotations into warnings, or suppress them from the rendered draft.
  - this directly affects renderer behavior, QA expectations, and acceptance criteria.
- Recommended direction: **Harden**
  - decide one of:
    1. add `Loops` and `Annotations` explicitly to the Phase 1 template, or
    2. state that both are rendered inside `Exceptions / warnings` and `Assumptions / open questions`.
  - make the product spec and UC use the same wording.

### [P1] The docs still do not agree on whether BRD markdown is rendered deterministically from the canonical spec or generated by the model again

- Evidence:
  - the product spec says Step 6 is “Render từ structured spec sang markdown theo template đã chọn” in [ai-brd-description-feature.md](../product/ai-brd-description-feature.md:251), and says templates are just two render views over one canonical data model in [ai-brd-description-feature.md](../product/ai-brd-description-feature.md:423).
  - but the same product spec also says `gpt-5.5` is used to “render BRD draft chất lượng cao” in [ai-brd-description-feature.md](../product/ai-brd-description-feature.md:440).
  - the backend architecture reinforces the ambiguous path by describing `services/render.py` as calling the provider for both Step 5 and Step 6 in [architecture-brd-backend.md](../scope/architecture-brd-backend.md:84).
  - UC-06 describes Step 9 as model -> structured spec, then markdown render from that spec in [UC-06-sinh-brd-tu-diagram.md](../use-cases/UC-06-sinh-brd-tu-diagram.md:33).
- Impact:
  - this is a core architectural ambiguity, not a wording nit.
  - if Step 6 is another free-form LLM pass, traceability, determinism, cost, and post-check scope all change materially.
  - if Step 6 is deterministic template rendering, the architecture and model-strategy docs should not imply a second generation pass.
- Recommended direction: **Refactor in place**
  - pick one contract now.
  - my recommendation: canonical spec is the only model-generated artifact; markdown is rendered deterministically from the spec in Phase 1.
  - reserve LLM rewriting/regeneration of sections for Phase 2 explicit flows.

### [P2] `architecture-brd-backend.md` says it “chốt deployment topology” but still leaves the actual hosting decision open

- Evidence:
  - the document intro says it “chốt deployment topology” in [architecture-brd-backend.md](../scope/architecture-brd-backend.md:3).
  - Section 7 still presents Fly.io and VPS as parallel options in [architecture-brd-backend.md](../scope/architecture-brd-backend.md:137).
  - Section 12 lists deployment target and production domain as unresolved questions in [architecture-brd-backend.md](../scope/architecture-brd-backend.md:187).
- Impact:
  - the document currently behaves more like an architecture option memo than a finalized backend architecture note.
  - this is acceptable, but the framing should be honest so downstream readers do not assume ops decisions are already made.
- Recommended direction: **Harden**
  - either commit to one Phase 1 hosting choice, or rename the document intent from “chốt deployment topology” to “đề xuất topology và quyết định còn mở”.

### [P2] The backend env-var contract is incomplete because local workflow depends on `BRD_PROVIDER`, but the config table never defines it

- Evidence:
  - local setup uses `BRD_PROVIDER=mock` in [architecture-brd-backend.md](../scope/architecture-brd-backend.md:167) and [architecture-brd-backend.md](../scope/architecture-brd-backend.md:172).
  - the env-var table in [architecture-brd-backend.md](../scope/architecture-brd-backend.md:123) does not include `BRD_PROVIDER`.
- Impact:
  - this creates a drift between setup instructions and actual configuration contract.
  - the first backend bootstrap PR will have to guess whether provider selection is env-driven, code-driven, or implicit from key presence.
- Recommended direction: **Harden**
  - add `BRD_PROVIDER` to the env-var table with allowed values such as `openai` and `mock`.
  - document precedence rules when provider is `openai` but key is missing.

### [P3] The v2 review snapshot is now partly stale because it still recommends creating docs that already exist

- Evidence:
  - review v2 says NX1 is to create `UC-06-sinh-brd-tu-diagram.md` in [2026-05-31-ai-brd-feature-doc-review-v2.md](2026-05-31-ai-brd-feature-doc-review-v2.md:120).
  - it also says NX3 could be a separate `architecture-brd-backend.md` doc in [2026-05-31-ai-brd-feature-doc-review-v2.md](2026-05-31-ai-brd-feature-doc-review-v2.md:136).
  - its disposition again recommends creating both docs in [2026-05-31-ai-brd-feature-doc-review-v2.md](2026-05-31-ai-brd-feature-doc-review-v2.md:173).

  - this is not dangerous to implementation, but it makes the review snapshot less trustworthy as a “latest truth”.
  - someone reading only v2 may think these artifacts are still missing.
- Recommended direction: **Keep as-is with addendum**
  - do not rewrite history, but add a short note or superseding link pointing to the newer artifacts and remaining unresolved issues.
