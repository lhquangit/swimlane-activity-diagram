# Review why the `Use Cases` screen shows `Theo rule (scaffold)` instead of AI generation

Date: 2026-06-12
Reviewer: Codex
Scope: persisted `Use Case` generation mode on the current local environment

## Conclusion

The AI generation feature exists in code, but your current local server configuration is not using
it for `Use Case` generation. The screen is telling the truth: it is currently running the
deterministic scaffold path, not the AI synthesis path.

## Evidence

1. Local `apps/api/.env` currently sets BRD OpenRouter config, but does not define any
   `USECASE_*` overrides:
   - `apps/api/.env` contains `BRD_PROVIDER=openrouter` and `BRD_OPENROUTER_API_KEY=...`
   - `apps/api/.env` does not contain `USECASE_PROVIDER` or `USECASE_GENERATION_MODE`
2. Backend config defaults `USECASE_GENERATION_MODE` to `deterministic` when it is not set in the
   environment, at `apps/api/app/config.py:81-87`.

## What `scaffold` means here

In this repository, `scaffold` means a starter draft generated by the deterministic builder, not by
the LLM. It is a rule-based structure used as a safe fallback or starter artifact.

The actual implementation is in `apps/api/app/usecases/deterministic_builder.py:62-155`, where the
system assembles generic intake / execution / coordination style use cases from the feature inputs.

## Why AI is not appearing

The missing piece is not the OpenRouter key itself. The missing piece is the effective generation
mode:

- `USECASE_GENERATION_MODE` is not set locally
- the backend therefore falls back to `deterministic`
- deterministic mode short-circuits AI generation entirely

So the correct mental model is:

- AI generation path exists
- local runtime currently disables it for `Use Case`
- the UI is exposing that disabled state, though the wording is still too internal

## Recommended direction

- Current state: behavior is technically correct, but the wording is still product-hostile.
- Main risks:
  - users interpret `scaffold` as a bug or missing feature instead of an intentional fallback mode
  - teams cannot tell whether AI is unavailable because of config, rollout mode, or provider health
- Recommended direction: Refactor in place
- Why now: the system finally became truthful, but the message is still not understandable to a
  normal product user.
